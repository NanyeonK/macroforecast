# Macrocast Full Option-Universe Implementation Roadmap

Goal
- Turn the current grammar-first package into a package that can eventually execute the full option universe explicitly represented in registry/path space.
- Preserve one path = one fully specified forecasting study.
- Preserve the split between representable choice space and current executable support while systematically promoting remaining registry-only/planned choices into operational support.

Status

### Phase C — Framework widening

### Phase D — Benchmark-family widening
Purpose
- continue adding benchmark executors independently of model executors

Status
- first custom benchmark bridge complete

What is complete
- `benchmark_family = custom_benchmark` is operational through a plugin-ready local Python callable bridge
- compiler preserves and validates `benchmark_config.plugin_path` plus `benchmark_config.callable_name`
- runtime loads the callable, executes it as the benchmark forecaster, and preserves the benchmark contract in manifest provenance

### Phase E — Evaluation and statistical tests
Status
- partially complete

What is complete
- `stat_test = dm` is operational
- `stat_test = cw` is operational
- runtime writes `stat_test_dm.json`
- runtime writes `stat_test_cw.json`
- manifest records `stat_test_spec` and `stat_test_file`

### Phase F — Importance layer
Status
- partially complete

What is complete
- `importance_method = minimal_importance` is operational
- runtime writes `importance_minimal.json`
- manifest records `importance_spec` and `importance_file`
- first operational support exists for:
  - one non-AR linear family: `ridge`
  - one tree family: `randomforest`

What remains
- broader minimal-importance coverage across more model families
- richer importance semantics for lagged-target routes
- `shap`

## 3. Ordered execution sequence

Recommended order from here:
1. Phase E continuation — broader comparison artifacts beyond DM/CW
2. Phase B continuation — broader preprocessing variants
3. Phase F continuation — broader importance coverage / SHAP
4. Phase G — real-time and multi-target widening

Reason
- the package now has a usable first interpretation layer
- the next high-value gains are broader benchmark/evaluation coverage and richer importance beyond the first slice
