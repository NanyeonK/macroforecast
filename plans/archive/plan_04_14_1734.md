# Macroeconomic Forecasting Research Package Development Plan

## A Fair and Extensible Benchmarking Framework for FRED-MD, FRED-QD, and FRED-SD Based Forecasting Studies

## 1. Project Overview

This project aims to develop a research-oriented Python package for macroeconomic forecasting using three standardized macro datasets: **FRED-MD, FRED-QD, and FRED-SD**. The package is not intended to be merely a model collection or a convenience toolkit. Its primary goal is to provide a **fair, reproducible, and extensible benchmarking framework** in which researchers can evaluate forecasting tools under controlled and comparable conditions.

More specifically, the package will allow users to hold some design choices fixed, vary others systematically, and assess how differences in model class, preprocessing, feature construction, validation design, and tuning strategy affect forecasting performance. The core philosophy is that a forecasting study should be represented as a **fully specified experimental path**, so that comparisons across tools are transparent and replicable. This design principle follows the idea that one experimental path should correspond to one complete forecasting study, rather than a loose collection of partially specified settings. 

The package is intended to support both methodological research and practical benchmark construction in macroeconomic forecasting. In particular, it should be useful for researchers who want to test whether a new forecasting tool genuinely improves performance, or whether apparent gains arise only from inconsistent preprocessing, data splits, benchmark choice, or evaluation design.

## 2. Motivation and Problem Statement

Macroeconomic forecasting studies often suffer from a common problem: models are compared under conditions that are not fully aligned. In many cases, differences in sample periods, horizon construction, information sets, lag structure, preprocessing rules, hyperparameter tuning, or benchmark definitions make it difficult to determine whether one method truly outperforms another.

This problem becomes more severe in data-rich forecasting settings such as those using FRED-MD and FRED-QD. Once the researcher introduces alternative target definitions, feature extraction methods, factor models, machine learning estimators, time-series cross-validation schemes, and interpretability methods, the design space expands rapidly. Without a clear grammar for what is fixed and what is varied, the resulting framework becomes difficult to maintain and even harder to interpret.

The present project addresses this issue by building a package in which the experimental structure is explicit. In this framework, each study is defined as a complete recipe. Discrete design choices are separated from numeric parameters, and fixed comparison axes are separated from sweep axes. This makes the framework suitable not only for forecasting execution, but also for structured benchmarking, ablation studies, and replication work. 

## 3. Main Objective

The main objective of this project is to build a package that enables researchers to evaluate macroeconomic forecasting tools under fair and reproducible conditions using FRED-MD, FRED-QD, and FRED-SD data.

The package will be designed to answer questions such as the following:

* Does a given forecasting model outperform standard benchmarks when the information set and evaluation protocol are fixed?
* How much of performance improvement comes from the model itself, rather than preprocessing or feature engineering?
* Are gains stable across forecast horizons, target variables, and sample periods?
* How sensitive are results to choices such as scaling, factor extraction, lag construction, validation strategy, and benchmark selection?
* Do variable importance results remain stable across model classes and experimental settings?

## 4. Core Design Principles

The package will be built around four core principles.

First, **one path equals one fully specified forecasting study**. Each run must correspond to a complete design, not a partially defined experiment.

Second, **enumerated choices and numeric or free parameters must be separated**. Discrete design decisions, such as data source, preprocessing method, model family, and evaluation metric, should be represented explicitly. Numeric values such as lag length, number of factors, or tuning budget should be stored in configuration fields rather than overloading the experimental path. 

Third, **fixed axes and sweep axes must be separated**. A valid comparison requires a clear distinction between elements that remain constant across models and elements that are intentionally varied.

Fourth, **the Python module tree and the experiment tree must be separated**. The internal codebase should remain simple and modular, while the experimental design space should be managed through taxonomies, registries, recipes, and run artifacts rather than through deeply entangled module logic. 

## 5. Scope of Version 1

To avoid excessive complexity at the initial stage, the first working version will deliberately restrict scope.

Version 1 will focus on **FRED-MD only**, using the **revised monthly information set**. The package will initially support **single-target point forecasting**, with forecast horizons such as **1, 3, 6, and 12 months ahead**. The first implementation will use an **expanding-window framework**, standard benchmark models such as **historical mean** and **AR selected by BIC**, and a small initial set of forecasting models such as **AR**, **Ridge/Lasso**, and **Random Forest**.

Preprocessing in Version 1 will be minimal and controlled. It will include only a small set of target transformations, predictor scaling rules, and lag or factor feature builders. Evaluation will focus on a compact set of metrics such as **MSFE**, **relative MSFE**, and **OOS R²**, together with a small number of statistical comparison tools such as **Diebold-Mariano type tests**.

Interpretability and advanced statistical testing will not be the center of the first release. Those components will be added only after the basic experimental pipeline is stable.

## 6. Planned Package Structure

The package will be organized so that code execution and research design remain clearly separated. The internal Python structure will remain minimal, while experimental flexibility will be handled by recipes and registries.

A simplified target structure is as follows:

```text
macrocast/
  data/
  preprocess/
  features/
  models/
  pipeline/
  evaluation/
  io/
  schemas/
  taxonomy/
  recipes/
  runs/
```

The long-run design will include a taxonomy-based research layer with components such as data definitions, target and predictor construction, preprocessing, training, evaluation, statistical tests, importance methods, and output provenance. The original design notes already suggest that a top-level structure based on `taxonomy/`, `recipes/`, and `runs/` is the most appropriate way to manage this complexity. 

## 7. Experimental Grammar

A central element of the package is the definition of a canonical experimental grammar.

The package will define a fixed order of layers for every study, for example:

`meta → data → target/x → preprocess → training → evaluation → statistical tests → importance → output/provenance`

Within this grammar, the study path will record only enumerated design choices. Numeric parameters will be stored in recipe files or configuration fields. This avoids excessively long or unstable path strings and keeps the experimental record readable. This approach is explicitly recommended in the design notes, where enum choices are placed in path segments and numeric parameters are kept in leaf configurations. 

## 8. Recipe-Based Study Design

Each forecasting study will be defined by a single YAML recipe. This recipe will contain all information necessary to reproduce the study: data source, target variable, horizons, preprocessing rules, feature construction, model type, validation method, benchmark choice, metrics, and output settings.

In this framework, a recipe becomes the unit of scientific comparison. A researcher can define one baseline recipe, modify one element, and then test whether the resulting performance difference is attributable to that element alone. This makes the package especially useful for ablation studies, replications, and structured benchmark suites.

## 9. Development Phases

The development will proceed in several stages.

### Phase 1. Experimental contract and schema design

The first phase will define the study contract, canonical layer order, recipe schema, and artifact schema. At this stage, the goal is not to support many models, but to establish a stable grammar for what constitutes a valid forecasting study.

### Phase 2. Minimal end-to-end pipeline

The second phase will implement a minimal end-to-end workflow using FRED-MD. This stage will include data loading, preprocessing, feature building, expanding-window forecasting, benchmark comparison, metric calculation, and artifact saving.

A successful completion of this phase means that one YAML recipe can produce one full run with reproducible outputs.

### Phase 3. Registry and taxonomy expansion

Once the minimal pipeline is stable, the package will introduce structured registries for data definitions, preprocessing choices, model families, evaluation metrics, and statistical tests. This will make the package easier to extend without destabilizing the core execution logic.

### Phase 4. Additional datasets and advanced evaluation

After the monthly FRED-MD workflow is stable, the package will extend to **FRED-QD** and then **FRED-SD**. Additional features such as conditional evaluation, model confidence set procedures, interpretability modules, grouped importance analysis, and advanced forecasting objects can be introduced at this stage.

### Phase 5. Replication and benchmark suite

In the final stage, the package will support paper-style replications, benchmark suites, and comparison studies. This will allow the package to serve not only as a forecasting engine but also as a macro forecasting research platform.

## 10. Expected Contributions

This project is expected to make three main contributions.

First, it will provide a **fair comparison framework** for macro forecasting studies. Instead of comparing methods under inconsistent settings, researchers will be able to evaluate tools under controlled conditions.

Second, it will provide a **reproducible study grammar**. By representing each forecasting study as a recipe-based experimental path, the package will reduce ambiguity and improve transparency in forecasting research.

Third, it will provide an **extensible benchmarking platform**. Because the package separates module logic from the experiment tree, new data sources, models, preprocessors, metrics, and interpretability methods can be added without redesigning the entire system.

## 11. Evaluation Criteria for the Package Itself

The package will be considered successful if it satisfies the following criteria.

A single recipe must be sufficient to run one complete forecasting study. Re-running the same recipe should yield identical or controlled reproducible results. Different models must be comparable under the same information set, sample split, benchmark, and metric definitions. Run artifacts must be clearly saved and sufficiently rich to support inspection, replication, and paper-ready reporting.

Most importantly, the package must allow a researcher to answer a simple but fundamental question: whether a forecasting tool improves performance because of the tool itself, rather than because of hidden differences in data treatment or evaluation design.

## 12. Risks and Mitigation

The main risk is excessive complexity. Because the design space is very large, there is a strong temptation to support multiple datasets, horizons, model classes, importance methods, and statistical tests too early. This would likely produce an unstable and difficult-to-maintain framework.

To mitigate this, the project will follow a strict staged development strategy. The first working version will deliberately limit the data source, task type, model family, and evaluation tools. Only after the minimal pipeline is stable will the framework be generalized.

A second risk is leakage or unfair comparison caused by inconsistent preprocessing or validation. This will be addressed by enforcing train-only fitting rules, recipe validation, and explicit comparison contracts.

A third risk is that the taxonomy becomes too elaborate before the execution engine is stable. To avoid this, the implementation will prioritize the working pipeline over registry completeness.

## 13. Conclusion

This project proposes the development of a research-oriented macroeconomic forecasting package that combines reproducibility, fairness, and extensibility. By focusing on experimental grammar first and model variety second, the package aims to provide a reliable platform for evaluating forecasting tools using FRED-MD, FRED-QD, and FRED-SD data.

The central idea is simple but important: a forecasting study should be fully specified, reproducible, and comparable. Once that is guaranteed, model innovation becomes scientifically interpretable. Without that structure, apparent forecasting gains are often difficult to trust.
